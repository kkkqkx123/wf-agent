# 工具描述管理架构设计文档

## 1. 设计目标

### 1.1 核心目标

1. **用户定义优先**: 工具描述完全由用户在工具定义时提供，系统不做任何猜测或生成
2. **工具模块自包含**: 工具模块提供完整的工具描述管理功能，不依赖外部集成
3. **保证工具存在性**: 通过工具注册器验证工具的存在性，避免引用不存在的工具
4. **直接提供原始数据**: 工具模块直接提供用户定义的原始数据，不做任何转换或增强
5. **简单直接**: 接口简单明了，职责单一，易于理解和使用

### 1.2 设计原则

1. **单一数据源**: 工具描述信息只存储在工具定义中，避免多处维护
2. **不做假设**: 不假设工具描述的格式、内容或用途
3. **不生成内容**: 不生成任何工具描述内容，只提供用户定义的内容
4. **不转换格式**: 不转换工具描述格式，只提供原始格式
5. **向后兼容**: 保持与现有工具定义的兼容性

## 2. 核心概念

### 2.1 工具定义的组成

工具定义包含以下核心部分：

1. **标识信息**
   - 工具ID（唯一标识符）
   - 工具名称（用于调用）
   - 工具类型（STATELESS/STATEFUL/REST/MCP）

2. **描述信息**
   - 工具描述（用户定义的完整描述，可以是任何格式）
   - 参数Schema（JSON Schema格式，定义参数结构）

3. **配置信息**
   - 工具配置（根据工具类型不同而不同）
   - 元数据（可选的分类、标签等）

### 2.2 工具描述的两种用途

#### 用途1：提示词中的工具介绍
- **数据来源**: 工具的`description`字段
- **使用方式**: 直接使用用户定义的描述文本
- **格式**: 用户定义的任何格式（纯文本、Markdown等）
- **处理**: 不做任何处理，直接使用

#### 用途2：LLM API的工具定义
- **数据来源**: 工具的`name`、`description`、`parameters`字段
- **使用方式**: 直接使用用户定义的数据
- **格式**: ToolSchema格式（name, description, parameters）
- **处理**: 不做任何处理，直接使用

### 2.3 关键认识

**重要**: 工具模块只负责存储和提供工具定义，不负责：
- 生成工具描述
- 转换工具描述格式
- 增强工具描述
- 应用描述模板
- 验证工具描述质量

所有这些功能都应该由使用工具模块的代码负责，而不是工具模块本身。

```
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                     工具注册层                                │
│  ┌──────────────────────────────────────────────────────┐  │
│  │           工具注册器 (ToolRegistry)                   │  │
│  │  - 验证工具定义的完整性                                │  │
│  │  - 存储工具定义和描述信息                              │  │
│  │  - 提供工具查询接口                                    │  │
│  └──────────────────────────────────────────────────────┘  │
└────────────────────┬────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                   工具模块边界                                │

                     ▼
┌─────────────────────────────────────────────────────────────┐
│                   使用层（外部模块）                          │
│  ┌──────────────────┐  ┌──────────────────┐                │

```

### 3.2 工具定义结构

#### 3.2.1 基础工具定义

```typescript
interface Tool {
  // 标识信息
  id: ID;
  name: string;
  type: ToolType;
  
  // 描述信息（用户定义）
  description: string;
  parameters: ToolParameterSchema;
  
  // 配置信息
  config?: ToolConfig;
  metadata?: ToolMetadata;
  
  // 检查点配置
  createCheckpoint?: boolean | 'before' | 'after' | 'both';
  checkpointDescriptionTemplate?: string;
}
```

**关键点**:
- `description`字段完全由用户定义，可以是任何格式、任何内容
- `parameters`字段是JSON Schema格式，定义参数结构
- 工具模块不验证`description`的内容或格式
- 工具模块不生成或修改`description`

#### 3.2.2 ToolSchema格式

```typescript
interface ToolSchema {
  name: string;
  description: string;
  parameters: ToolParameterSchema;
}
```

**关键点**:
- ToolSchema是工具定义的简化版本
- 只包含LLM API需要的字段
- 直接从Tool定义中提取，不做任何修改

### 3.3 工具注册器

#### 3.3.1 工具注册器的职责

1. **验证工具定义**
   - 验证必需字段是否存在（id, name, type, description, parameters）
   - 验证字段类型是否正确
   - 验证参数Schema的有效性（JSON Schema格式）
   - **不验证**description的内容或格式

2. **存储工具定义**
   - 存储完整的工具定义
   - 建立索引（按名称、类型）
   - 支持快速查询

3. **提供查询接口**
   - 按ID查询工具
   - 按名称查询工具
   - 按类型查询工具
   - 列出所有工具

4. **保证工具存在性**
   - 在注册时验证工具ID的唯一性
   - 在查询时验证工具的存在性
   - 提供工具列表接口

#### 3.3.2 工具注册器的验证规则

1. **必需字段验证**
   - 工具ID必须存在且唯一
   - 工具名称必须存在
   - 工具类型必须有效
   - 工具描述必须存在（但不验证内容）
   - 参数Schema必须存在且符合JSON Schema规范

2. **可选字段验证**
   - 如果提供config，必须符合工具类型的要求
   - 如果提供metadata，必须是有效的对象

3. **参数Schema验证**
   - 必须符合JSON Schema规范
   - 必需参数必须在properties中定义
   - 参数类型必须有效

### 3.4 工具模块的边界

#### 3.4.1 工具模块提供什么

1. **工具注册和查询**
   - 注册工具定义
   - 查询工具定义
   - 列出所有工具

2. **工具定义的原始数据**
   - 返回完整的Tool定义
   - 返回ToolSchema格式（简化版本）
   - 返回工具的description字段
   - 返回工具的parameters字段

#### 3.4.2 工具模块不提供什么

1. **不提供描述生成**
   - 不生成工具描述
   - 不增强工具描述
   - 不修改工具描述

2. **不提供格式转换**
   - 不转换描述格式
   - 不转换参数格式
   - 不转换为特定LLM提供商格式

3. **不提供模板功能**
   - 不支持描述模板
   - 不支持变量替换
   - 不支持动态描述

4. **不提供质量验证**
   - 不验证描述质量
   - 不提供改进建议
   - 不检查描述完整性

## 4. 数据流设计

### 4.1 工具注册流程

```
用户定义工具
    ↓
工具注册器验证
    ↓
验证通过？
    ├─ 是 → 存储工具定义
    │         ↓
    │       建立索引
    │         ↓
    │       注册成功
    │
    └─ 否 → 返回错误信息
```

### 4.2 工具查询流程

```
请求工具定义
    ↓
工具注册器接收请求
    ↓
从存储中获取工具定义
    ↓
工具存在？
    ├─ 是 → 返回工具定义
    │
    └─ 否 → 返回错误信息
```

### 4.3 工具使用流程（外部模块）

```
外部模块需要使用工具
    ↓
从工具注册器获取工具定义
    ↓
根据用途选择使用方式
    ├─ 用于提示词 → 使用description字段
    └─ 用于LLM API → 使用ToolSchema格式
    ↓
外部模块自行处理（转换、增强等）
```

## 5. 外部模块的职责

### 5.1 提示词生成模块

**职责**:
- 从工具注册器获取工具定义
- 使用工具的`description`字段
- 根据需要组织提示词格式
- 可以添加额外的说明或示例

**示例**:
```typescript
const tool = toolRegistry.get('calculator');
const prompt = `可用工具:\n- ${tool.name}: ${tool.description}`;
```

### 5.2 LLM调用模块

**职责**:
- 从工具注册器获取工具定义
- 将工具定义转换为ToolSchema格式
- 根据LLM提供商要求转换格式
- 调用LLM API

**示例**:
```typescript
const tool = toolRegistry.get('calculator');
const toolSchema = {
  name: tool.name,
  description: tool.description,
  parameters: tool.parameters
};
// 使用现有的格式转换器
const formatted = convertToolsByProvider([toolSchema], provider);
```

### 5.3 动态工具处理模块

**职责**:
- 接收动态工具配置
- 从工具注册器获取工具定义
- 应用描述模板（如果需要）
- 生成工具列表

**示例**:
```typescript
const tools = dynamicTools.toolIds.map(id => toolRegistry.get(id));
if (dynamicTools.descriptionTemplate) {
  tools.forEach(tool => {
    tool.description = applyTemplate(dynamicTools.descriptionTemplate, tool);
  });
}
```

## 6. 错误处理

### 6.1 工具不存在错误

**场景**: 引用的工具ID不存在

**处理方式**:
1. 在工具注册时验证工具ID的唯一性
2. 在获取工具时验证工具的存在性
3. 如果工具不存在，返回明确的错误信息
4. 错误信息包含：工具ID、可用工具列表

### 6.2 工具定义无效错误

**场景**: 工具定义缺少必需字段或字段类型错误

**处理方式**:
1. 在工具注册时验证工具定义的有效性
2. 如果定义无效，返回明确的错误信息
3. 错误信息包含：无效的字段、错误原因、修复建议

### 6.3 参数Schema无效错误

**场景**: 参数Schema不符合JSON Schema规范

**处理方式**:
1. 在工具注册时验证参数Schema的有效性
2. 如果Schema无效，返回明确的错误信息
3. 错误信息包含：无效的部分、错误原因、修复建议

## 7. 向后兼容性

必须替换所有旧的实现，不能为了向后兼容保留任何会破坏新的设计的代码实现。当前项目处于开发阶段，不存在任何兼容性要求
