# Graph 模块集成测试设计

## 测试设计原则

1. **场景划分**: 按功能场景划分测试文件，避免混合
2. **端到端**: 测试完整的执行流程，验证组件协作
3. **边界覆盖**: 重点测试边界条件和异常情况
4. **依赖隔离**: 使用 Mock 隔离外部依赖（LLM、工具等）

---

## 测试文件划分

### 1. 工作流注册与预处理测试
**文件**: `workflow-registration-and-preprocessing.test.ts`

**测试场景**:
- 基础工作流注册流程
  - 简单工作流（START -> LLM -> END）的注册和预处理
  - 验证 GraphRegistry 中是否正确存储预处理后的图
  - 验证工作流元数据完整性

- 复杂工作流注册流程
  - 包含多种节点类型的工作流注册
  - 验证所有节点正确展开

- 重复注册处理
  - 相同 ID 工作流重复注册应抛出错误
  - 验证错误信息准确性

- 批量注册
  - 多个工作流同时注册
  - 验证批量处理正确性

- 工作流注销
  - 单工作流注销
  - 批量注销
  - 注销后查询应返回 undefined

### 2. 图构建与验证测试
**文件**: `graph-building-and-validation.test.ts`

**测试场景**:
- 基础图构建
  - 从工作流定义构建 GraphData
  - 验证节点和边正确转换
  - 验证邻接表正确生成

- 环检测
  - 无环工作流应通过验证
  - 有环工作流应检测出环并报告错误
  - 复杂环（嵌套环）检测

- 可达性分析
  - 所有节点应从 START 可达
  - 所有节点应能到达 END
  - 不可达节点应被识别
  - 死胡同节点应被识别

- START/END 节点约束
  - START 节点不能有入边
  - END 节点不能有出边
  - 多个 START 节点应报错

- Fork/Join 配对验证
  - 正确配对的 Fork/Join 应通过
  - 未配对的 Fork 应报错
  - 未配对的 Join 应报错
  - Fork 和 Join 的 pathId 不匹配应报错
  - Fork 无法到达 Join 应报错

- 孤立节点检测
  - 孤立节点应被识别并报告

## 测试基础设施

### Mock 组件
- **MockLLMService**: 模拟 LLM 调用，返回预定义响应
- **MockToolService**: 模拟工具执行
- **MockEventManager**: 捕获和验证事件
- **MockCheckpointStorage**: 模拟检查点存储

### 工具函数
- **createSimpleWorkflow**: 创建简单测试工作流
- **createComplexWorkflow**: 创建复杂测试工作流
- **waitForThreadCompletion**: 等待线程完成执行
- **expectThreadResult**: 验证线程执行结果

### 测试数据
- **sampleWorkflows**: 预定义测试工作流集合
- **sampleNodes**: 各类节点配置示例

---

## 测试执行建议

1. **并行执行**: 各测试文件可并行执行
2. **隔离环境**: 每个测试文件独立设置和清理
3. **CI 集成**: 在持续集成中运行全部测试
4. **覆盖率**: 目标代码覆盖率 > 80%
