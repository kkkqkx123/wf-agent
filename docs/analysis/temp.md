目前已经提供了 ​context-configs.ts​ ，但是现有实现是针对某个执行实例的完整消息数组的，这可能存在问题。agent使用子agent时、workflow创建复杂分支时都会需要同时维护多个消息数组上下文(或临时裁剪、拼接并传递给子执行实例)。
分析agent、workflow是否需要支持同时维护多个message数组，以避免batch需要同时服务于临时数据、子模块的场景，让batch专注于当前运行实例的上下文编辑

还有一点，当前subgraph内置了消息裁剪机制，这是之前为了优化临时消息创建而专门设计的。如果迁移到多消息管理，就可以使用更优雅的方式传递，避免混合在节点配置中。不过需要考虑如何把消息管理嵌入workflow结构定义中。
目前我的想法是context处理节点可以指定产生的新消息的id(语义化字符串)，支持任意使用消息的节点(包括llm节点、subgraph、agent-loop、context处理节点[用于二次编辑])直接使用。agent层我还没想好怎样设计

简化llm node的prompt部分，仅提供prompt参数，直接传递消息数组。其他全部作为message层面的拼接。这样可以减少字符串序列化开销，并简化处理逻辑
