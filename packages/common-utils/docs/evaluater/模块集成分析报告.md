# evalutor 模块集成分析报告

## 1. 概览

`packages/common-utils/src/evalutor` 目录提供表达式 DSL 解析、评估、条件评估、安全校验等能力。  
当前通过 `index.ts` 导出 **11 组 API**，外部实际使用的仅 **4 组**，其余 7 组仅在 evalutor 内部或测试中引用。

## 2. 当前已使用的模块

| 导出模块 | 外部消费方 |
|---------|-----------|
| `expressionEvaluator` | `sdk/core/hooks/executor.ts`, `sdk/workflow/execution/.../variable-handler.ts` |
| `conditionEvaluator` | `route-handler.ts`, `loop-end-handler.ts`, `workflow-execution-coordinator.ts`, `triggers/matcher.ts`, `workflow-navigator.ts` |
| `validateExpression` | `route-validator.ts`, `loop-end-validator.ts`, `hook-validator.ts`, `trigger-validator.ts` |
| `resolvePath` | `packages/common-utils/src/template/template-renderer.ts`（同包内部使用） |

## 4. 集成建议

### 第1档：应当集成的（有明确的消费场景）

#### 4.1 `ExpressionCompiler` → 集成到 `expressionEvaluator` 和 `conditionEvaluator`

- **现状**：每次 `expressionEvaluator.evaluate()` 调用都重新 parse + evaluate，无缓存。
- **集成方案**：`expressionEvaluator.evaluate()` 在 DSL parse 后调用 `compiler.compile()` 缓存 AST，后续相同表达式命中缓存直接评估；`conditionEvaluator.evaluate()` 也可复用预编译表达式。
- **改造位置**：
  - `expression-evaluator.ts` → `evaluate()` 方法
  - `condition-evaluator.ts` → `evaluate()` 方法

#### 4.2 `DependencyManager` / `VariableChangeTracker` → 集成到执行引擎的路由/循环/触发器

- **现状**：路由条件、循环条件、触发器条件每次被调用时完全重新评估，即使输入变量未变化。
- **集成方案**：在 `route-handler.ts`、`loop-end-handler.ts`、`triggers/matcher.ts` 中注册表达式到 `DependencyManager`，仅在依赖的变量发生变化时才重算条件。变量通过 `VariableChangeTracker` 统一跟踪变更。
- **改造位置**：这些 handler 的 `evaluateCondition()` 或等效方法，改为从 `DependencyManager` 获取结果。

#### 4.3 `setPath` / `pathExists` → 集成到 `variable-handler.ts` 和 `context-builder.ts`

- **现状**：`variable-handler.ts` 设置变量时直接操作对象属性，`context-builder.ts` 构建评估上下文时逐层手动赋值。
- **集成方案**：使用 `setPath` 替换手写的深浅拷贝和赋值逻辑（内置安全校验）；`pathExists` 用于前置检查路径是否存在后再取值。
- **改造位置**：
  - `variable-handler.ts` → `execute()` 中的变量赋值逻辑
  - `context-builder.ts` → 上下文对象构建逻辑

### 第2档：按需集成（开发工具/调试场景）

#### 4.4 `visualizeAST` / `traceEvaluation` / `formatTrace` → 集成到 SDK 调试模式

- **现状**：表达式评估失败时只有简单错误消息，没有 AST 级别的调试信息。
- **集成方案**：在 `expressionEvaluator` 增加 `debug` 模式 flag；启用时通过 `traceEvaluation` 记录评估过程；`conditionEvaluator.evaluate()` 返回失败时调用 `visualizeAST(expression)` 输出 AST 结构辅助排查。
- **改造位置**：
  - `expression-evaluator.ts` → 构造参数增加 `debug?: boolean`
  - `condition-evaluator.ts` → 条件不满足时的日志分支

#### 4.5 `extractAllMetadata` / `findNodeAtPosition` → 集成到工作流设计器 API（远期）

- **现状**：暂无编辑器/IDE 集成需求。
- **集成路径**：当需要实现表达式编辑器时，`findNodeAtPosition` 可用于实现光标定位到 AST 节点；`extractComments` 可用于表达式文档提示。
- **改造位置**：保留为备用，不接入生产运行时。

### 第3档：建议保留但不需要外部集成

#### 4.6 `validatePath` / `validateArrayIndex` / `validateValueType` / `SECURITY_CONFIG`

- 已通过 `expressionEvaluator.evaluate()` 和 `validateExpression()` 对外提供安全屏障。
- **不建议**再暴露直接调用——安全策略应由上层 `validateExpression()` 统一封装，下层路径检查由 `expressionEvaluator` 内部委托。
- 对外入口为 `validateExpression()`（安全+语法双重校验），隔离细节。

#### 4.7 `validateComparisonTypes` / `validateArrayMethodResult` / `isValidNumber` / `isArray`

- 纯内部类型校验辅助函数，无外部集成价值。
- 通过 `expressionEvaluator.evaluate()` 内部的类型安全检查对外生效。

#### 4.8 `dslValidate` / `tokenizeExpression`

- `dslValidate` 的校验能力目前通过 `validateExpression` 整合暴露（安全 + 语法双重校验），无需独立导出。
- `tokenizeExpression` 为底层词法分析函数，已通过 `dslParse` 间接服务所有消费者。

## 5. 优先级建议

| 优先级 | 模块 | 集成目标 | 影响范围 |
|--------|------|---------|---------|
| **P0** | `ExpressionCompiler` | 集成到 `expressionEvaluator.evaluate()` | 性能提升，低风险，修改集中 |
| **P0** | `setPath` | 集成到 `variable-handler.ts` | 变量设置安全化，替换现有裸写逻辑 |
| **P1** | `DependencyManager` | 集成到 route / loop-end / trigger / matcher | 条件评估性能优化，需设计状态管理方案 |
| **P1** | `pathExists` | 集成到条件前置检查 | 减少冗余评估 |
| **P2** | `traceEvaluation` / `visualizeAST` | 表达式调试模式 | 开发体验提升，非功能必需 |
| **P3** | `ast-metadata` 系列 | 工作流设计器表达式编辑器 | 远期 IDE 支持，当前不接入 |
| 无需集成 | security-validator / type-validator / dslValidate / tokenizeExpression | — | 已在内部间接生效 |
