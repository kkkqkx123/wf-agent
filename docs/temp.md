我指的检查点不是为了保证消息数组的稳定性，而是为了减少相关操作的开销。
你之前在node type中的ContextProcessorNodeConfig设计的createNewBatch需要彻底删除，因为消息数组自身的逻辑必须保证不会丢失任何信息。此外，我的想法是把消息的管理从llm分离出来，把所有可用操作(例如ContextProcessorNodeConfig的操作类型)也一起放在单独的类型定义中，保证绝对的统一来源。这里类型定义仅音乐

关于操作类型：对于插入，可以考虑不再支持-1，始终为中间插入，尾插专门由append类型来实现。append不需要开始新批次，其他则始终需要开始新批次。
其中clear实际上不会引入额外的数组元素拷贝开销，而其他操作实际上都会引入大量开销(因为需要把旧的批次的内容拷贝为新的批次)。

由于消息数组直接操作检查点管理显然是有问题的，可以考虑另一个思路：
在类型定义(新的单独类型定义)中添加批次快照数组。对于clear，快照为空，因为没有额外拷贝开销；对于其他操作(除append根本不需要新批次)，直接把消息数组深拷贝为快照，这样不再需要维护过度延长的数组(否则不仅需要拷贝元素，还会大幅延长数组)

## 核心发现

### 2. 作用域管理体系

**多层级架构**：
- 变量作用域：global → thread → local → loop
- 工具可见性作用域：THREAD → WORKFLOW → global
- 独立的管理器负责不同作用域类型

**作用域切换机制**：
- 子图执行时自动切换作用域
- 循环执行维护独立的作用域栈
- 工具可见性随作用域动态更新

### 3. 主要问题识别

#### 🟡 中风险问题
2. **作用域切换不同步**：变量和消息作用域切换独立进行
3. **工具声明时序问题**：批量声明存在延迟窗口
4. **消息隔离性不足**：Fork/Join 场景下机制不明确

#### ⚠️ 低风险问题
1. **声明历史不完整**：缺乏完整性验证机制
2. **模板变量作用域混淆**：解析时可能访问错误作用域

## 关键风险点

### 并发执行风险
- 消息数组和标记映射可能处于不一致状态
- 工具可见性更新可能在线程销毁后触发
- Fork/Join 操作的消息合并策略不明确

### 数据完整性风险
- 过滤操作可能意外删除系统关键消息
- 声明消息缺乏完整性验证，可能被伪造

### 作用域一致性风险
- 变量作用域切换与消息作用域更新不同步
- 子图切换时工具可见性存在时间窗口
- 循环作用域中的消息与变量历史混合存储

## 改进建议

### 立即实施（高优先级）
1. **实施读写锁机制**：保护 `ConversationManager` 的关键操作
2. **添加状态一致性检查**：定期验证消息数组和索引映射的一致性
3. **实施压缩备份机制**：为消息压缩操作提供回滚能力

### 中期实施（中优先级）
1. **作用域事务管理**：确保变量、消息、工具作用域的原子性切换
2. **智能压缩策略**：保护系统消息、工具声明和变量引用
3. **声明消息签名**：防止工具可见性声明被伪造或篡改

### 长期优化（低优先级）
1. **Fork/Join 消息隔离**：明确子线程消息继承和合并策略
2. **审计日志系统**：记录所有消息操作和作用域切换
3. **性能监控**：监控消息操作对系统性能的影响
